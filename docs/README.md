# LevelUp Study - ドキュメント設計 (Documentation Guide)

この `docs/` ディレクトリは、プロジェクトの計画から設計、実装までの流れを整理するためのドキュメント群です。
それぞれのフォルダ（階層）が持つ「役割」と「用途」は以下の通りです。

---

## 📂 ディレクトリ構成と役割

### 1. `planning/` (企画・ロードマップ)
「何を、いつ作るのか」を管理するディレクション領域です。
- **`01_Features_and_Roadmap.md`**: 実装すべき機能の一覧、ステータス（考案/設計/TODO/実装済み）のカンバン的な役割を果たします。
- **用途**: 新しい機能のアイデアを思いついたら、まずはここの「💡 考案」セクションに追記します。

### 2. `templates/` (テンプレート)
ドキュメントの書式を統一するための雛形置き場です。
- **`Requirements_Design_Template.md`**: 機能の提案〜要件定義〜設計までをひとまとめに書く用のフォーマットです。
- **用途**: 新しい機能の設計を書く際は、このテンプレートをコピーして `features/` フォルダ下に配置します。

### 3. `features/` (機能ごとの詳細設計)
個々の機能に関する具体的な要件と設計書（仕様書）を置く領域です。
- ファイルの例: `StudyTimer_Requirement.md`, `GachaSystem_Design.md` など
- **用途**: `planning/` の中で「よし、これを作ろう」と決まった機能について、どのようなUIで、どんなAPIを叩くのかを `templates` に従って詳細に書き出します。AI（私）に実装を依頼する際、ここにある設計書を指定して「この通りに作って」と指示すると精度が飛躍的に上がります。

### 4. `architecture/` (全体アーキテクチャ・基盤設計)
アプリ全体の構造や、プラットフォームの枠組みなど「全体基盤」に関わる設計領域です。
- **`01_Overview.md`**: 現在採用しているKMPやGoのモノレポ構成の全体像が書かれています。
- **用途**: 新しく参画したメンバーやAIが、プロジェクトの「マクロな構造」を理解するために最初に読むドキュメントです。

---

## 🔄 開発の基本的なフロー
1. **思いつく** ➔ `planning/01_Features_and_Roadmap.md` の「考案」に追加。
2. **設計する** ➔ `templates/` からテンプレートをコピーし、`features/〇〇.md` として詳細を記載。
3. **実装する** ➔ AIに「『features/〇〇.md』を読んで実装して」と指示し、作成開始。
4. **完了** ➔ `planning/01_Features_and_Roadmap.md` のステータスを「実装済み」に移動する。
