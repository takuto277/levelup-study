# Workflow: /task-init

task_name = $ARGUMENTS

以下の手順でタスクの実行計画書を作成してください。

1. ユーザーから依頼されたタスクの概要を整理する
2. `docs/tasks/` ディレクトリに実行計画書を作成する
3. ファイル名は `YYYYMMDD-<task_name>.md` 形式にする
4. 以下のテンプレートに沿って記入する。特に「実装詳細」セクションでは、変更箇所のコード引用と、その変更を行う具体的な理由（なぜその処理が必要か、なぜその実装方法を選ぶのか）を詳しく記述してください。

```markdown
# タスク: <タスク名>

| 項目 | 値 |
|------|-----|
| 作成日 | YYYY-MM-DD |
| ステータス | 計画中 / 進行中 / 完了 / 中止 |
| 担当 | AI / 人間 / 共同 |

## 概要
タスクの目的と背景を1〜3行で記述。

## 要件
- [ ] 要件1
- [ ] 要件2

## 影響範囲
変更対象のファイル・モジュールを列挙する。

## 実装手順
1. ステップ1
2. ステップ2

## 実装詳細
ここでは、具体的なコードの変更内容とその理由を記述します。

### 1. [ファイル名または機能名]
- **変更内容**:
  ```[言語]
  // 変更前後のイメージや、追加するロジックの抜粋
  ```
- **理由**: なぜこの変更が必要なのか、この実装にすることでどのような課題が解決されるのかを記述。

## リスク・注意点
- 既知のリスクや制約事項

## テスト計画
- テスト方法と確認ポイント

## 結果・振り返り
完了後に記入する。実際の変更内容、発生した問題、学びなど。
```

5. 作成が完了したら、計画書のパスを報告する
